= Crie e gerencie políticas de backup do Microsoft SQL Server no NetApp Backup and Recovery
:hardbreaks:
:allow-uri-read: 
:icons: font
:imagesdir: ./media/


[role="lead"]
No NetApp Backup and Recovery, crie suas próprias políticas de backup do Microsoft SQL Server que controlam a frequência do backup, o horário em que o backup é realizado e o número de arquivos de backup que são retidos.


NOTE: Algumas dessas opções e seções de configuração não estão disponíveis para todas as cargas de trabalho.

Se você importar recursos do SnapCenter, poderá encontrar algumas diferenças entre as políticas usadas no SnapCenter e aquelas usadas no NetApp Backup and Recovery.  Verlink:reference-policy-differences-snapcenter.html["Diferenças de política entre SnapCenter e NetApp Backup and Recovery"] .

Você pode atingir os seguintes objetivos relacionados às políticas:

* Criar uma política de instantâneo local
* Crie uma política para replicação para armazenamento secundário
* Crie uma política para configurações de armazenamento de objetos
* Configurar configurações avançadas de política
* Editar políticas
* Excluir políticas




== Ver políticas

. No menu NetApp Backup and Recovery , selecione *Políticas*.
. Analise os detalhes da política. Por exemplo:
+
** *Carga de trabalho*: Exemplos incluem Microsoft SQL Server, ONTAP Volumes, VMware, KVM, Hyper-V, Oracle Database ou Kubernetes.
** *Tipo de backup*: Exemplos incluem backup completo e backup de log.
** *Arquitetura*: Exemplos incluem snapshot local, fan-out, cascateamento, disco para disco e disco para armazenamento de objetos.
** *Recursos protegidos*: mostra quantos recursos do total de recursos naquela carga de trabalho estão protegidos.
** *Proteção contra ransomware*: mostra se a política inclui bloqueio de snapshot no snapshot local, bloqueio de snapshot no armazenamento secundário ou bloqueio de DataLock no armazenamento de objetos.






== Criar uma política

Você pode criar políticas que controlam seus snapshots locais, replicações para armazenamento secundário e backups para armazenamento de objetos. Parte da sua estratégia 3-2-1 envolve a criação de um snapshot das instâncias, bancos de dados, aplicativos ou máquinas virtuais no sistema de armazenamento *primário*.

*Função necessária do NetApp Console * Visualizador de armazenamento, superadministrador de backup e recuperação, administrador de backup de backup e recuperação.  Aprenda sobrelink:reference-roles.html["Funções e privilégios de backup e recuperação"] . https://docs.netapp.com/us-en/console-setup-admin/reference-iam-predefined-roles.html["Saiba mais sobre as funções de acesso do NetApp Console para todos os serviços"^] .

.Antes de começar
Se você planeja replicar para armazenamento secundário e deseja usar o bloqueio de snapshot em snapshots locais ou em armazenamento secundário ONTAP remoto, primeiro precisa inicializar o relógio de conformidade ONTAP no nível do cluster.  Este é um requisito para habilitar o bloqueio de snapshot na política.

Para obter instruções sobre como fazer isso, consulte https://docs.netapp.com/us-en/ontap/snaplock/initialize-complianceclock-task.html["Inicializar o relógio de conformidade no ONTAP"^] .

Para obter informações sobre bloqueio de instantâneo em geral, consulte https://docs.netapp.com/us-en/ontap/snaplock/snapshot-lock-concept.html["Bloqueio de instantâneo no ONTAP"^] .

.Passos
. No menu NetApp Backup and Recovery , selecione *Políticas*.
. Na página Políticas, selecione *Criar nova política*.
+
A página de Policies é exibida.

. Insira as informações na seção *Detalhes*:
+
** Tipo de carga de trabalho: selecione *Microsoft SQL Server*.
** Insira um nome de política.
** Selecione um agente do Console na lista *Agente*.


. Insira as informações na seção *Arquitetura de backup*. Selecione o fluxo de dados para o backup na lista:
+
** *Fanout 3-2-1*: storage primário (disco) para storage secundário (disco) para nuvem (armazenamento de objetos). Cria múltiplas cópias de dados em diferentes sistemas de storage, como ONTAP para ONTAP e ONTAP para configurações de armazenamento de objetos. Isso pode ser um armazenamento de objetos de hyperscaler de nuvem ou um armazenamento de objetos privado. Melhor para proteção de dados ideal e recuperação de desastres. Esta opção não está disponível para Amazon FSx for NetApp ONTAP.
** *Cascata 3-2-1*: storage primário (disco) para storage secundário (disco) e storage primário (disco) para storage de nuvem (armazenamento de objetos). Isso pode ser um storage de objetos de hyperscaler de nuvem ou um storage de objetos privado, como StorageGRID. Isso cria uma cadeia de replicação de dados em vários sistemas para garantir redundância e confiabilidade. Esta opção não está disponível para Amazon FSx for NetApp ONTAP.
** *Disco para disco*: storage primário (disco) para storage secundário (disco). A estratégia de proteção de dados ONTAP para ONTAP replica dados entre dois sistemas ONTAP para garantir alta disponibilidade e recuperação de desastres. Isso geralmente é feito usando SnapMirror, que oferece suporte tanto à replicação síncrona quanto à replicação assíncrona. Esse método mantém seus dados atualizados e disponíveis em diferentes locais, proporcionando forte proteção de dados.
** *Armazenamento de disco para objeto*: Do storage primário (disco) para a nuvem (armazenamento de objetos). Isso replica dados de um sistema ONTAP para um sistema de storage de objetos. Isso pode ser um storage de objetos de hyperscaler de nuvem ou um storage de objetos privado, como StorageGRID. Esse método é ideal para retenção de dados a longo prazo e arquivamento. Esta opção não está disponível para Amazon FSx for NetApp ONTAP.
** *Fanout de disco para disco*: storage primário (disco) para storage secundário (disco) e storage primário (disco) para storage secundário (disco). Você pode configurar várias configurações secundárias para a opção de fanout de disco para disco.
** *Instantâneos locais*: Instantâneo local no volume selecionado. Isso cria cópias somente leitura, ponto no tempo, dos volumes de produção onde suas cargas de trabalho estão sendo executadas. Você pode usar instantâneos locais para se recuperar de perda de dados ou corrupção, bem como para criar backups para fins de recuperação de desastres.


. Forneça informações para a seção *Configurações de instantâneo local*:
+
** Selecione a opção *Adicionar agendamento* para selecionar o agendamento ou agendamentos de instantâneos.  Você pode ter no máximo 5 agendamentos.
** *Frequência do instantâneo*: selecione a frequência: horária, diária, semanal, mensal ou anual.  A frequência anual não está disponível para cargas de trabalho do Kubernetes.
** *Retenção de instantâneos*: insira o número de instantâneos a serem mantidos.
** *Habilitar backup de logs*: Habilite esta opção para fazer backup dos logs e defina a frequência e a retenção dos backups de logs. Para isso, você já deve ter configurado um backup de logs. Consulte link:br-start-configure.html["Configurar diretórios de log"].


. Forneça informações para a seção *Configurações secundárias* (replicação para storage secundário):
+
** *Backup*: Selecione a frequência: horária, diária, semanal, mensal ou anual.
** *Destino do backup*: Selecione o sistema de destino no armazenamento secundário para o backup.
** *Período de bloqueio do Snapshot*: selecione se deseja ativar os snapshots à prova de adulteração. Se ativado, insira o número de dias, meses ou anos que deseja bloquear o snapshot.
** *Transferir para o armazenamento secundário*: Escolha quando transferir os snapshots para o storage secundário.
+
*** A opção * Agendamento de transferência ONTAP - Em linha* é selecionada por padrão e indica que os instantâneos são transferidos para o sistema de armazenamento secundário imediatamente.  Você não precisa agendar o backup.


** *Usar relacionamento secundário existente do SnapMirror e SyncMirror*: habilite esta opção para usar o relacionamento existente do SnapMirror ou SyncMirror para transferir snapshots para o cluster de destino.


. Forneça informações para a seção *Configurações do armazenamento de objetos* (backup para o storage de objetos):
+

NOTE: Os campos que aparecem diferem dependendo do provedor e da arquitetura selecionada.

+
** *Provedor*: Selecione o provedor para seu armazenamento de objetos e insira as credenciais nos campos apropriados (os campos de credenciais variam dependendo do provedor).
** *Destino de backup*: selecione um storage de objetos registrado e verifique se ele está acessível em seu ambiente de backup.
** *IPspace*: Selecione o IPspace a ser usado para as operações de backup.  Isso é útil se você tiver vários IPspaces e quiser controlar qual deles será usado para backups.
** *Configurações de agendamento*: selecione o agendamento que foi definido para os instantâneos locais.  Você pode remover uma programação, mas não pode adicionar uma porque as programações são definidas de acordo com as programações de instantâneos locais.
** *Cópias de retenção*: Insira o número de snapshots a serem mantidos para cada agendamento.
** *Agendamento de transferência para armazenamento de objetos*: escolha o agendamento de transferência do ONTAP para fazer backup dos dados no storage de objetos.
** *Coloque seus backups em camadas do armazenamento de objetos para o armazenamento de arquivamento*: se você optar por colocar os backups em camadas para o armazenamento de arquivamento (por exemplo, AWS Glacier), selecione a opção de camada e o número de dias para arquivamento.






=== Configurar configurações avançadas na política

[%collapsible]
====
Você pode opcionalmente configurar opções avançadas na política. Você pode usar essas opções para todas as arquiteturas de backup e destinos de storage. As opções avançadas disponíveis dependem da carga de trabalho que você seleciona na parte superior da página, portanto, algumas opções descritas aqui podem não se aplicar a todas as cargas de trabalho.

.Passos
. No menu NetApp Backup and Recovery , selecione *Políticas*.
. Na página Políticas, selecione *Criar nova política*.
. Na seção *Política > Configurações avançadas*, selecione o menu *Selecionar ação avançada* para escolher em uma lista de configurações avançadas.
. Habilite qualquer uma das configurações que você deseja visualizar ou alterar e selecione *Aceitar*.
. Forneça as seguintes informações:
+
** *SnapMirror formato de volume e instantâneo*: Escolha uma das seguintes opções:
+
*** *Usar formato de nome personalizado para cópia do instantâneo*: escolha um esquema de nomenclatura para os instantâneos. Se deixado em branco, um carimbo de data/hora é adicionado ao final de cada nome de instantâneo.
*** *Fornecer formato de volume SnapMirror*: especifique um prefixo, um sufixo ou ambos para modificar o nome padrão do volume SnapMirror. Por padrão, um volume SnapMirror herda o nome do volume de origem.


** *Copy only backup*: Escolha copy only backup (um tipo de Microsoft SQL Server backup) se precisar fazer backup de seus recursos usando outro aplicativo de backup.
** *Configurações do grupo de disponibilidade*: selecione as réplicas de backup preferenciais ou especifique uma réplica em particular. Essa configuração é útil se você tiver um grupo de disponibilidade do SQL Server e quiser controlar qual réplica será usada para backups.
** *Habilitar snapshot consistente com a VM*: selecione se deseja habilitar snapshots consistentes com a VM. Isso garante que os snapshots recém-criados sejam consistentes com o estado da máquina virtual no momento do snapshot. Isso é útil para garantir que os backups possam ser restaurados com sucesso e que os dados estejam em um estado consistente. Isso não se aplica a snapshots existentes. Para workloads VMware, você pode habilitar essa opção para frequências de política específicas, exceto por hora.
** *Verificação de backup*: selecione se deseja ativar a verificação de backup e se deseja que ela seja executada imediatamente ou posteriormente. Esse recurso garante que os backups sejam válidos e possam ser restaurados com sucesso. Recomendamos que você ative essa opção para garantir a integridade dos seus backups. Por padrão, a verificação de backup é executada a partir do storage secundário, se o storage secundário estiver configurado. Se o storage secundário não estiver configurado, a verificação de backup será executada a partir do storage primário.
+
Além disso, configure as seguintes opções:

+
*** Verificação *Diária*, *Semanal*, *Mensal* ou *Anual*: Se você escolher *Mais tarde* como verificação de backup, selecione a frequência da verificação de backup.  Isso garante que os backups sejam verificados regularmente quanto à integridade e possam ser restaurados com sucesso.
*** *Rótulos de backup*: Escolha um nível de backup em sua política cujos backups você deseja verificar. Dependendo da sua configuração *Verificar no secundário*, a lista de níveis virá do seu agendamento primário ou secundário. Os backups horários e de log não aparecem, pois não são elegíveis para verificação.
*** *Verificação de consistência do banco de dados*: Opcionalmente, selecione limites a serem aplicados às verificações de consistência do banco de dados antes do backup.
*** *Verificar backups de logs*: selecione se deseja verificar os backups de logs para garantir que sejam válidos e possam ser restaurados. Selecione um ou mais servidores de verificação. Se você escolheu disco a disco ou 3-2-1, selecione também o local de storage para verificação.


** * Formato de volume e instantâneo do SnapMirror *: Opcionalmente, insira seu próprio nome de instantâneo em uma política que controla os backups para cargas de trabalho do Microsoft SQL Server. Insira o formato e o texto personalizado. Se você optar por fazer backup no armazenamento secundário, também poderá adicionar um prefixo e sufixo de volume do SnapMirror .
** *Taxa máxima de transferência*: Para não definir um limite para utilização de largura de banda, selecione *Ilimitado*. Se desejar limitar a taxa de transferência, selecione *Limitado* e defina a largura de banda entre 1 e 1.000 Mbps alocada para o upload de backups para o storage de objetos. Por padrão, ONTAP pode usar uma quantidade ilimitada de largura de banda para transferir os dados de backup dos volumes do sistema para storage de objetos. Se o tráfego de backup afetar as cargas de trabalho, reduza a largura de banda para as transferências.
** *Repetições de backup*: Para repetir a tarefa em caso de falha ou interrupção, selecione *Habilitar repetições de tarefa durante falha*. Insira o número máximo de repetições de tarefas de snapshot e backup e o intervalo de tempo entre as repetições. O número de repetições deve ser inferior a 10.
+

TIP: Se a frequência do snapshot for definida como 1 hora, o atraso máximo, juntamente com a contagem de novas tentativas, não deverá exceder 45 minutos.

** *Verificação de ransomware*: selecione se deseja habilitar a verificação de ransomware em cada bucket. Isso requer bloqueio do DataLock no armazenamento de objetos. Insira a frequência da verificação em dias. Esta opção se aplica ao armazenamento de objetos da AWS e do Microsoft Azure. Observe que esta opção pode incorrer em custos adicionais, dependendo do provedor de nuvem.
** *Notificação*: Escolha se deseja ativar as notificações por e-mail para operações de backup. Você pode selecionar quais eventos acionam uma notificação — por exemplo, quando um backup é bem-sucedido, falha ou é concluído com avisos.

== Editar uma política

Você pode editar a arquitetura de backup, a frequência de backup, a política de retenção e outras configurações de uma política. Para políticas de cargas de trabalho do Kubernetes, você pode editar apenas as configurações de agendamento e retenção.

Você pode adicionar outro nível de proteção ao editar uma política, mas não pode remover um nível de proteção.  Por exemplo, se a política estiver protegendo apenas instantâneos locais, você poderá adicionar replicação ao armazenamento secundário ou backups ao armazenamento de objetos.  Se você tiver snapshots e replicação locais, poderá adicionar armazenamento de objetos.  No entanto, se você tiver snapshots locais, replicação e armazenamento de objetos, não poderá remover um desses níveis.

Se estiver editando uma política que faz backup no armazenamento de objetos, você pode habilitar o arquivamento.

Se você importou recursos do SnapCenter, poderá encontrar algumas diferenças entre as políticas usadas no SnapCenter e aquelas usadas no NetApp Backup and Recovery.  Verlink:reference-policy-differences-snapcenter.html["Diferenças de política entre SnapCenter e NetApp Backup and Recovery"] .

.Função necessária do NetApp Console
Superadministrador de backup e recuperação. https://docs.netapp.com/us-en/console-setup-admin/reference-iam-predefined-roles.html["Saiba mais sobre as funções de acesso do NetApp Console para todos os serviços"^] .

.Passos
. No NetApp Console, acesse *Proteção* > *Backup e Recuperação*.
. Selecione a opção *Políticas*.
. Selecione a política que você deseja editar.
. Selecione as *Ações*image:icon-action.png["Ícone de ações"] ícone e selecione *Editar*.




== Excluir uma política

Você pode excluir uma política se não precisar mais dela.


TIP: Não é possível excluir uma política associada a uma carga de trabalho.

.Passos
. No Console, vá para *Proteção* > *Backup e Recuperação*.
. Selecione a opção *Políticas*.
. Selecione a política que você deseja excluir.
. Selecione as *Ações*image:icon-action.png["Ícone de ações"] ícone e selecione *Excluir*.
. Confirme a ação e selecione *Excluir*.
